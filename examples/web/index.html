QDRV browser output test harness

This example is a browser-side harness for checking what QDRV outputs look like through web APIs. It runs the qdrv-decode-wasm tone mapper (tone_map_frame) on a synthetic delivery-tier PQ frame, using real QDRV dynamic metadata, and shares its renderer with the WebCodecs/ImageDecoder/browser-image path below.

Full browser pipeline (experimental)

Load a delivery-tier .qdrv32 file directly, or an IVF stream produced with qdrv mux --format ivf, to run the browser-side QDRV path (metadata extraction to decoded pixels or metadata preview to WebGPU/wasm tone map), reusing the target-display values above. Direct .qdrv32 still-picture payloads are wrapped as AVIF for browser still-image decoding; if the browser rejects that QDRV AV1 profile, the harness renders a metadata preview through the same tone mapper. IVF streams use VideoDecoder.
